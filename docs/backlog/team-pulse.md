# Пульс команд — раздел Staff CRM (backlog)

**Статус:** в бэклоге, возвращаемся после допила основного функционала.

---

## Контекст

Еженедельные встречи с лидами сейчас ведутся через плоскую таблицу — она
нечитаема по мере роста. Пришли к выводу что нужны не статусы, а **реестр
эскалаций** + **аналитика пульса команд** для меня.

Лиды приносят только то, что требует моего вмешательства (Confluence,
одна страница). Аналитику веду я сам в Staff CRM.

---

## Структура раздела

### Статичная карточка команды

- Название, лид, DPO, стрим
- Цели команды на квартал (2–3 пункта, чекбокс «достигли/нет»)
- Ключевые риски (3 строки), обновляются редко
- Раунды/ритуалы: даты следующих PI planning, ретро, демо
- Ссылки: Confluence, Jira, Miro (URL[])

### Еженедельный снэпшот

Вношу после встречи с лидом.

- Неделя (ISO-неделя, прячется/проставляется автоматически)
- Статус — 🟢 / 🟡 / 🔴 (моя субъективная оценка)
- `tailIndex` — 0..10, плотность хвостов (субъективно)
- Эскалации: счётчик за неделю + «главная эскалация одной строкой»
  (опционально); полный реестр остаётся в Confluence
- Тип главной эскалации: `decision` / `resource` / `communication`
- Заметка — одно предложение, свободный текст

### Производные метрики (считаются автоматически)

- Сколько недель подряд в текущем статусе (`streak`)
- Частота эскалаций за последние 4 недели и 12 недель
- Тренд `tailIndex` (slope линейной регрессии за 12 недель) → ↘ / → / ↗

## Сценарии использования

- **Перед 1-1 с лидом** — открыл карточку, вспомнил контекст
- **Квартальный обзор** — паттерны по всем командам
- **Опережающий сигнал** — команда 3+ недели в жёлтом → пора предметно
  разговаривать

---

## Формат еженедельной встречи (переосмысленный)

Встреча делится на два режима:

1. **Разблокировать** — пробежка по реестру эскалаций (Confluence)
2. **Сверить курс** — что идёт хорошо, что хочет попробовать лид,
   где нужно мнение а не помощь

## Архитектура (предварительно)

### Хранение

Отдельная Supabase-таблица `team_pulse` (по образцу `teams`/`projects`):

```sql
create table public.team_pulse (
  id          uuid primary key default gen_random_uuid(),
  owner_id    uuid not null references auth.users(id) on delete cascade,
  payload     jsonb not null default '{}'::jsonb,
  created_at  timestamptz not null default now(),
  updated_at  timestamptz not null default now()
);
-- payload: { teamId, weekStart, status, tailIndex, escalations, escalationKind, note }
-- индекс: unique(owner_id, (payload->>'teamId'), (payload->>'weekStart'))
```

Снэпшоты — отдельные записи, не массив в payload команды. Иначе каждое
обновление будет переписывать весь массив через SyncQueue.

### UI

- **Главный экран `/teams`** — мини-индикатор пульса в строке команды:
  текущий status-чип + streak «🟡 3w» + крошечный SVG-спарклайн (24px) tailIndex
- **Страница команды `/teams/:id`** — статичная шапка + timeline снэпшотов
  + три KPI (streak, частота эскалаций 4w/12w, slope tailIndex) +
  кнопка «Снэпшот за неделю N» (модалка с авто-prefill из прошлой недели)
- **Отчёт `/pulse`** (опционально, для квартального обзора) — heatmap
  «команды × недели» (4 × 12 ячеек)

Без chart-библиотек — SVG-спарклайны вручную, ~20 строк.

## Что отложить / не делать на старте

- Автоматический подсчёт «хвостов» из Jira/Confluence — соблазн, но
  интеграции отдельный мир. Ручной ввод = честная метрика.
- Шаринг с лидом. Это **моя** аналитика. Если когда-то понадобится —
  toggle «видно лиду».
- AI-summary недели. Сначала пусть данные накопятся (≥ 3 месяца),
  потом подумаем.
